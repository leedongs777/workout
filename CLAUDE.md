## 2. 검증/테스트 워크플로 (신뢰성 확보된 방식)
편집 후 반드시 이 순서로 검증:
1. **문법 검사**: `<script>` 블록을 추출해 `node --check`.
   - 예: HTML에서 `<script>…</script>` 안 JS를 파일로 뽑아 `node --check file.js`.
2. **로직 유닛 테스트**: 함수/상수를 brace-match로 추출해 mock state로 `eval` 실행.
   - 헬퍼 패턴: 이름으로 함수 본문을 뽑는 `grab()`, `const`를 brace-match로 뽑는 `gc()`.
3. **UI 렌더링 확인**(가능한 환경일 때): 헤드리스 브라우저 스크린샷 등으로 확인. 과거엔 `wkhtmltoimage`(한국어 폰트 NanumGothic) + `cairosvg`를 썼음.
   - ⚠️ `wkhtmltoimage`는 **구형 webkit이라 CSS Grid 지원이 약함** → 달력 등 grid 레이아웃 미리보기가 실제와 다르게(겹치거나 세로 stack) 나옴. 실제 iOS/안드 Safari에선 정상. **grid는 그 도구로 시각 검증 불가**로 간주.
4. 편집 완료 후: `index.html`에서 `<script>` 추출해 `node --check`로 문법 확인 → 로직 테스트 → (가능하면) 렌더 확인.

## 3. 부상 기반 안전 프로그래밍 (조건부 규칙 — 사용자가 고른 부상에만 적용)

> ⚠️ **이력 주의**: 예전 이 문서는 "백세이프(척추 축부하 금지)"를 **모든 사용자에 대한 절대 규칙**으로 못박고 있었다. 그것은 **특정 개인의 신체 조건을 전역화**한 것이었고, **2026-07 결정으로 폐기**했다. 되돌리지 말 것.

### 3.1 동작 원리 — "금기"가 아니라 "주의"

> ⚠️ **2026-07 재설계.** 예전에는 `CONTRA_EX`(금기)로 운동을 **제외**했다. 이는 진료지침과 어긋난다.
> 요통(ACP 2017), 슬개대퇴통증(APTA/JOSPT 2019), 회전근개(JOSPT 2025) 모두
> ① 운동이 1차 치료이고 ② 특정 동작을 금지할 근거는 없으며 ③ **통증 반응에 따라 조절**하라고 권고한다.
> 따라서 지금은 `CAUTION_EX`(주의)이며 **제외하지 않는다.** 되돌리지 말 것.

- `CAUTION_EX = {운동명: [관절…]}`. 부상 부위와 겹치면 그 운동은 **후보에서 뒤로 밀린다**(제외 아님).
- `slotCands()`가 정렬한다: `caution=false` 먼저 → 같은 등급이면 `_HEAVY`(바벨 컴파운드)를 뒤로 → 그다음 원래 순서(안정 정렬).
  - 두 번째 키가 필요한 이유: 어깨 부상자의 `수직 밀기` 슬롯은 **후보가 전부 주의 대상**이라, 없으면 가장 무거운 바벨 오버헤드 프레스가 기본값이 된다.
  - 부상이 없으면 `caution`이 전부 false라 **원래 순서가 그대로 유지**된다(바벨 백 스쿼트가 1순위).
- 결과: **모든 부상 조합에서 슬롯이 사라지지 않는다**(주의 = 후순위, 제외 아님). 부상 없을 때 default 프로그램의 활성 슬롯은 현재 25개이며 전부 채워진다.
- 후보 순서 정의: **`options`(기구) → `fb`(기본 맨몸) → `alts`(안전 대안)**. `alts`를 `options`에 넣으면 **부상 없는 사용자의 기본 동작까지 바뀌므로 금지**.
- 관절 키: `knee`, `shoulder`, `ankle`, `lower_back`, `wrist`, `neck`.
- 주의 운동은 **기구 선택 버튼에 점선 테두리 + "주의" 배지**로 표시되고, 고르면 토스트로 안내한다. **선택은 사용자 몫이다.**
- 안전 원칙 문구도 "제외했어요"가 아니라 **"뒤로 미뤘어요, 금지된 동작은 아니에요"** 로 쓴다.

### 3.2 기본값 = 표준 프로그래밍
- **부상을 선택하지 않은 사용자**는 바벨 백 스쿼트 / 컨벤셔널·루마니안 데드리프트 / 바벨 벤치 프레스 / 바벨 오버헤드 프레스 / 바벨 벤트오버 로우를 **1순위로 받는다**(장비 보유 시).
- **`허리 (디스크/수술)`를 고른 사용자**만 위 운동들이 `lower_back` 태그로 빠지고, 등받이 지지 머신 · 힙힌지/둔근 중심 · 맥길 코어(데드버그·버드독·사이드 플랭크) 구성으로 대체된다.

### 3.3 유지할 것
- 모든 동작 큐(cue)에 "허리 중립 유지, 통증 시 즉시 중단" 취지 반영(모든 사용자 공통 안전 문구).
- `CAUTION_EX` 태그는 **의학적 검증을 받지 않은 초안**이다. 태그를 바꾸려면 근거를 남기고 운영자 확인을 받을 것.
- 부상/지병 관련 **안내 문구는 프로필 조건부**로만 노출한다. (과거 버그: 특정 개인의 병력을 전제한 문장이 모든 사용자에게 노출됐음)

## 3.5 근거·출처 원칙 (신뢰도의 핵심)

> 사용자가 이 앱을 믿고 운동하다 다칠 수 있다. **근거 없는 수치를 단정적으로 제시하지 말 것.**

- 코드의 **`SRC` 레지스트리**가 사용자에게 제시하는 수치의 1차 출처를 담는다. 새 수치를 넣으면 반드시 여기에 근거를 추가하고 화면에 출처를 노출한다.
- **확인된 출처 (2026-07)**
  - 최대심박 추정 `208 − 0.7 × 나이` — Tanaka H, Monahan KD, Seals DR. *Age-predicted maximal heart rate revisited.* J Am Coll Cardiol. 2001;37(1):153-156. 351개 연구 메타분석. (기존 `220 − 나이`는 Fox 1971이며 젊은 층 과대·고령층 과소 추정, 70세에서 약 10 bpm 차이)
  - 강도 분류 **중강도 65~75 %HRmax / 고강도 76~96 %HRmax** + 대화 검사 + RPE 0~10(중강도 3~4, 고강도 5~7) — ACSM, *Tips for Monitoring Aerobic Exercise Intensity*.
- **심박존 5단계는 학술 표준이 아니다.** 지구력 실무자 710명 중 47%가 5존을 쓰지만(Sci Rep, 2025), 같은 논문이 *"no consensus has so far been established"* 라고 명시한다. 연구용 기준은 문턱(LT1/LT2, VT1/VT2) 기반 3존이며 실험실 측정이 필요하다.
  - 그래서 우리 존 경계는 **ACSM에서 유도**한다: Z1 50~64 / **Z2 65~75(중강도)** / Z3 76~85 / Z4 86~96 / Z5 96~100.
  - **Z3/Z4 경계 85%는 임의값**이다. 화면에 그렇게 명시되어 있다. 지우지 말 것.
- **고정 %HRmax는 개인차가 크다.** Zone 2 지표의 개인간 CV 6~29%, 같은 %HRmax에서 실제 파워 CV 22.7~27.6% (PMC11986187). 그래서 **대화 검사를 심박수보다 앞에 노출**한다.
- **고강도(Zone 3+) 인터벌은 기본 처방에서 제외**한다. `cardioDetail(s)`가 `체력 향상`·`근육 만들기`를 고른 사용자에게만 허용하고, 그 외에는 Zone 2로 낮춘다.
- **부상·지병 관련 확인된 출처 (2026-07)**
  - 요통: ACP 진료지침(2017) — 운동이 1차 치료. 특정 동작 금지 권고 없음. `https://www.acpjournals.org/doi/10.7326/M16-2367`
  - 슬개대퇴통증: APTA/JOSPT(2019) — 스쿼트·무릎 신전은 금기가 아니며 통증 없는 범위에서 점진적으로. `https://www.jospt.org/doi/10.2519/jospt.2019.0302`
  - 회전근개: JOSPT(2025) — 운동 치료가 1차. 통증 범위 회피 여부는 근거 미확립. `https://www.jospt.org/doi/10.2519/jospt.2025.13182`
  - 고혈압: ACSM — 안정시 180/110mmHg 이상이면 운동 전 의사 상담, 저항 운동 중 발살바 금지. `https://acsm.org/exercise-for-the-prevention-and-treatment-of-hypertension/`
- **`SRC` 목록은 설정 화면 "근거와 출처" 섹션에 노출**된다. 사용자가 직접 링크로 확인할 수 있어야 한다.
- ⚠️ **아직 출처가 없는 것**: 80개 운동의 자세 큐, 세트×반복 처방, `CAUTION_EX` 태그의 **개별 운동 배정**, `goalMods` 가중치, 시간 배분 비율. 전부 LLM 생성이며 **의학적 검토를 받지 않았다.** 설정 화면에도 그렇게 명시되어 있다. 순차적으로 검증할 것(§7.5).

## 4. 앱 아키텍처 (index.html 내부)

### 4.1 저장/상태
- 저장 계층: `window.storage → localStorage → memory`. 키 **`workoutPlanner:v1`**.
- `persist()`(디바운스 120ms), `touch()`(=_ts 갱신 + persist + **디바운스된 Sync.push 500ms**). Firebase 설정 키 `workoutPlanner:fbcfg`.
- **`freshState()`**: 기본 state를 만드는 단일 소스. state 초기화·`resetAll`·`googleSignOut` 모두 이걸 쓴다(필드 누락 방지).
- **`nowTs()` = `Date.now()+_clockSkew`**: `_ts`는 이걸로 찍는다. `_clockSkew`는 Firebase `.info/serverTimeOffset`로 세팅(미연결 시 0) — 기기 시계 오차로 최신 편집이 밀리는 것 방지.
- ⚠️ **인증/동기화 안전장치(2026-07, Firebase 연결 사용자에 한정)**:
  - **로그아웃/전체초기화는 로컬 데이터를 클리어**한다(`freshState`) — 같은 기기 다음 사용자에게 이전 기록·프로필이 새지 않게. `resetAll`은 `Sync.signOut()`도 먼저 호출(빈 state가 클라우드를 덮거나 세션 잔존으로 게이트 우회하는 것 방지).
  - **`Auth.onSignedIn`은 동의 없는 계정 로그인을 통과시키지 않는다** — `state.consent`가 없으면 게이트를 닫지 않고 동의 화면으로. (예전 버그: attach 레이스로 신규 로그인이 약관 동의를 건너뜀.)
  - **`applyRemote`는 날짜 키 필드(`log`/`steps`/`stepStart`/`stepMs`)를 로컬·원격 병합**한다(문서 전체 LWW라 두 기기가 다른 날짜를 편집하면 소실되던 것 완화). 배열(doneDays 등)은 여전히 LWW.
  - 🔴 **실데이터 없는 새 기기는 `state._ts`를 `now`로 찍지 말 것**(부팅·attach 둘 다). `_ts`가 0/undefined여야 첫 동기화에서 `remote.updatedAt >= state._ts`가 참이 되어 **원격을 다운로드**한다. now로 찍으면 빈 state가 원격보다 최신으로 보여 **클라우드 기록을 빈 문서로 덮어씀(소실)**. `_ts`는 실제 편집 시 `touch()`가 stamp. (2026-07, 되돌리지 말 것.)
  - **인증 게이트는 `state.consent` 없으면 통과 금지** — `Auth.google`의 기존계정(isNewUser=false) 경로도 `!state.consent`면 동의 화면으로(우회 금지). `onSignedIn`도 동일.
  - ⚠️ 이 경로들은 **Firebase+브라우저 없이는 실행 테스트 불가** — 상태머신 로직만 node로 검증됨. 실제 로그인/동기화는 라이브 테스트 필요(§7 출시 전 항목).
- **state 필드**: `equipment[]`, `includeBodyweight`, `includeFloor`, `workoutDays[]`(요일 인덱스, 0=일), `sessionMinutes`, `startDate`, `restDays[]`, `doneDays[]`, `log{}`, `steps{}`(날짜별 단계 완료), `stepStart{}`(단계 시작 타임스탬프), `stepMs{}`(단계 소요 ms), `slotCustom{}`, `cardioChoice`, `cardioMin{}`(세션 인덱스별 유산소 분 override, 없으면 추천값 `sessionTimes().cardioAuto`), `authPassed`, `authProvider`, `consent`, `profile`, `onboarded`, `_ts`.
- `resetAll`과 state 기본값 모두 `workoutDays`, `sessionMinutes` 포함.
- ⚠️ **기본값은 중립적이어야 한다.** `workoutDays`는 **주 4일 중립값 `DEFAULT_WORKOUT_DAYS=[1,3,5,6]`(월·수·금·토)**, `DEFAULT_EQUIP`은 **빈 배열(기구 없음)**로 통일했다(2026-07, 과거 주 6일·27종은 특정 개인 일정·헬스장이라 폐기 — 되돌리지 말 것). 기구가 비면 맨몸이 자동 ON이라 첫 진입·둘러보기는 맨몸 플랜으로 시작하고, 사용자는 온보딩/설정에서 자기 기구를 직접 고른다. 온보딩을 마친 사용자는 자기 값을 쓴다.

### 4.2 스케줄 엔진 (요일 기반)
- `wDays()` = `state.workoutDays`(없으면 `DEFAULT_WORKOUT_DAYS`). **기본값은 이 상수 한 곳에서만 정의**한다(state 초기화·resetAll·weekTotal·toggle·설정 화면 모두 이걸 참조). 하드코딩 `[0,2,3,4,5,6]` 되살리지 말 것.
- `isRestDow(d)` = 해당 요일이 workoutDays에 없으면 정기 휴식.
- `buildSchedule`, `sessionFor` 모두 `isRestDow` 사용(휴식 = isRestDow || restDays). `sessionFor`는 startDate 이전엔 null. **null(시작 전)과 rest는 다르다** — 렌더에서 반드시 분리(`renderHome`도 분리함, 예전엔 null을 휴식으로 오표기).
- **`clampStartDate()`**: 시작일이 오늘보다 미래면 오늘로 당긴다. `load()`·동기화 `applyRemote()`·init 세 곳에서 호출(다중기기 sync로 미래 시작일이 들어와 스케줄이 깨지는 것 방어). 되돌리지 말 것.
- **통계 분모 주의**: `weekTotal()`(=`wDays().length`)은 **`programFor` 빈도 티어 전용 구조값**이다. 홈 "이번 주 완료 N/M"의 분모는 수동 휴식·시작일을 반영하는 **`weekPlanned()`**(이번 주 비휴식일을 `sessionFor`로 카운트)를 쓴다 — 둘을 혼동하면 N/M이 어긋난다(N>M 등).
- **로테이션은 고정 6개가 아니라 `programFor()`가 돌려주는 순환표**다: startDate부터 비휴식일을 카운트(`rot`)해 `prog[rot % prog.length]`. 반환값에 `pos`(순환 내 위치)·`len`(순환 길이) 포함 → "순환 N/M" 라벨과 "M일 순환" 문구가 동적. **`SESSIONS[e.session]`은 여전히 유효**(session은 라이브러리 인덱스).
- 하드코딩된 요일 참조 금지 — 항상 `isRestDow`. **하드코딩 `%6`·"/6"·"6일" 금지** — `programLen()`/`entry.len` 사용.

### 4.2a 빈도 티어 (`programFor` 1차 분기, 2026-07)

> ⚠️ **핵심 원칙(2026-07 재설계).** 순환 구조는 **먼저 주당 운동 일수(`weekTotal()`)가 정하고**, 목표는 그 위에 `goalMods`로 얹는다. 운동생리학적으로 **저빈도일수록 전신(full-body)**이 옳기 때문(적은 날에 부위 분할을 얇게 펴면 부위별 훈련 빈도가 급감). 되돌리지 말 것.

- `programFor()`가 `weekTotal()`로 분기한다:
  - **1~2일 → `[FB_A, FB_B]`** (전신 A/B). 둘이 함께 전신을 완전 커버 → 주 2일이면 각 부위 2회, 1일이면 주마다 A·B 교대.
  - **3일 → `[FB_A, FB_B, 8]`** (전신 A/B + 전신 순환 S8) → 부위별 주 3회.
  - **4일 → `[UP_A, 0, UP_B, 3]`** (상체A·하체A·상체B·하체B) → 상/하 각 2회. 4일 표준.
  - **5일+ → `PROGRAMS[programKey()]`** (아래 목표별 스플릿).
- **저빈도에서도 목표는 반영된다**: 구조는 빈도가 정하지만 `goalMods`(세트·반복·유산소·휴식·교정)는 세션 안에서 그대로 작동한다. 자세교정 목표는 FB_B/UP_A/UP_B에 든 후면삼각·페이스풀·말린어깨 교정 슬롯이 세트 +1을 받는다.
- **새 세션 S11~S14는 기존 슬롯을 참조(`_sl(si,oi)`)해 구성**한다(SESSIONS 리터럴 뒤 `SESSIONS.push`). 슬롯 큐·옵션·부상 대안이 원본과 100% 동일하게 상속되고, 로그·커스텀 키는 (세션 인덱스, 슬롯 위치)라 공유 참조여도 충돌 없다. **새 종목 큐를 새로 쓰지 말고 기존 슬롯을 참조**할 것.
  - 제목은 `sessionCat`이 부위를 title로 판정하므로 전신 세션은 `전신 A/B`(하체/상체/당기기/밀기/둔근/균형 단어 금지 → `full`로 잡힘), 상체 세션은 `상체 A/B · …`(→`upper`)로 둔다.

### 4.2b 목표별 프로그램 (`PROGRAMS` / `programFor`, 5일+ 전용, 2026-07)
- `PROGRAMS[key]` = 운동일 순환에 배치할 **세션 인덱스 배열**. **주 5일 이상일 때만** `programKey()`가 **1순위 목표**로 고른다(2순위는 스플릿 불변, `goalMods` 다이얼로만 반영 — 사용자 결정). 4일 이하는 §4.2a 빈도 티어가 우선.
- 5개 목표 전부 매핑 완료(유지=default):
  - `default:[0,1,2,3,4,5]` 균형(유지/무목표 폴백).
  - `hyper:[0,1,4,3,6,7]` 상하체 2회/주, **전용 유산소일 없음**(유산소는 각 날 마무리 존2만).
  - `diet:[8,2,1,5,4,2,3,2]` 근력4:유산소4(1:1) — 대사 컨디셔닝 + 근육 보존.
  - `endur:[9,2,0,1,9,5,2,4]` 유산소 우위(5:3) — 인터벌 중심, 근력 보조.
  - `posture:[1,0,10,4,3,2]` 교정 강조 — 당기기/하체A/교정집중/밀기/하체B/유산소.
- **설정 "구성" 목록·"내 프로필"** 은 `SESSIONS` 전체가 아니라 `programFor()`를 순회해 실제 순환만 보여줄 것.
- ⚠️ 순환 비율(1:1, 5:3 등)은 원리(간섭효과·근육 보존·심폐 우위)에 기반하나 **정확한 일수 비율은 검증된 표준이 아니라 구성값**. 슬롯 처방과 동급으로 "미검증"(§3.5) 취급.

- S0 하체A·둔근 / S1 상체 당기기·자세교정 / S2 유산소·모빌리티 / S3 하체B·좌우균형 / S4 상체 밀기·자세교정 / S5 유산소·코어·교정 / **S6 상체 당기기 B·등 두께(hyper) / S7 상체 밀기 B·어깨 중심(hyper) / S8 전신 순환·대사 컨디셔닝(diet) / S9 인터벌 컨디셔닝·유산소(endur) / S10 자세교정 집중·상체 교정(posture)**.

- **볼륨 밸런싱 패스(2026-07)**: 프로그램별 주간 볼륨을 실측(sessionExercises×effVol)해 과다·중복을 정리함.

  - S1에서 흉추 신전, S4에서 거북목 제거(교정은 S10·S1·S6에 남김) — 자세교정 프로그램 턱당기기 3→2회, 교정 48→38.

  - diet 순환에서 S5 1회만, posture 순환에서 S5 제외 — 코어·교정 중복 축소.
  - ⚠️ 교정 슬롯은 강도일(S1/S4/S6/S7)에도 있어야 **2순위 자세교정** 사용자(전용 프로그램 아님)가 교정을 받는다. 그래서 자세교정 1순위 프로그램에선 교정 볼륨이 태생적으로 높음(의도된 편중). 주 2회 교정 빈도는 교정운동 특성상 오히려 적절해 남김.
  - **종목 다양화(2026-07)**: ① S9의 첫 슬롯을 전신(스텝)→메인 하체(고블릿 스쿼트)로 교체 — endur가 S9·S2를 각 2회 돌아 스텝업이 4일 나오던 것을 2일로. ② posture 순환을 [1,0,10,4,3,2]로 재배열 — 교정 무거운 S1·S10이 연달지 않게 사이에 하체A. ③ **B세션(S6·S7·S10)의 옵션 순서를 A세션과 반대로** — S10 리어 델트 플라이·밴드 페이스풀 먼저(vs S1 리버스 펙덱·케이블 페이스풀), S6 리어 델트 플라이 먼저(vs S1), S7 덤벨 오버헤드 익스텐션 먼저(vs S4 푸시다운, 장두 강조로 자극도 분화). 옵션 순서가 기본 선택을 정하므로(slotCands 안정 정렬) **의도적 순서 차이를 되돌리지 말 것**. ④ diet 순환 [8,2,1,5,4,2,3,2] — 근·유 완전 교차 유지하면서 팔로프 나오는 날(1·4·7)이 연달지 않게 밀기(5일)·하체B(7일) 순서 배치.
- 각 slot: `{role, sets, reps, options:[{name, eq[], cue, reps?}], fb, alts?:[…]}`.
  - `reps`는 **슬롯 단위 기본값**이고, 운동별로 `reps`를 주면 그게 우선(`defReps(slot, r)`). 유지형(월 싯 `30–45초`)이 그 예. 같은 슬롯의 기구 운동은 영향받지 않는다.
  - `alts`는 **부상 시에만 쓰이는 안전 대안**(3.1 참고). `options`에 넣지 말 것.
- 슬롯의 실제 동작 선택은 `sessionExercises`가 `slotCands(slot)`의 **첫 후보**(사용자가 `slotPick`으로 고른 게 있으면 그것)로 정한다. **금기 제외 없음** — 부상 주의 운동은 `slotCands`가 뒤로 밀 뿐(제외 아님, §3.1).
- `isBW(eq)` = `eq.every(e=>e==='mat')` — 빈 배열도 `true`. **`eq`를 빼먹으면 장비가 필요한 운동이 맨몸으로 새어나간다**(과거 `밴드 레그 컬` 버그). 맨몸 동작의 큐에도 장비를 언급하지 말 것.
- **맨몸(무장비) 동작 28종**으로 기구가 하나도 없어도 바닥 포함 시 슬롯이 모두 채워진다(default 활성 25슬롯 기준, 현재값).
- **운동 명칭은 정식/검색 가능한 표준 명칭만.** (예전에 "회전근개 외회전/케이블 외회전"→**페이스 풀**로 교체. "A + B" 식 결합 명칭 금지.)
- **장비 없이 못 하는 동작은 대체(fb)로 두지 말 것.** 벤치가 필요한 `체어 스쿼트`는 `eq:['bench']` 옵션이고, 무장비 대체는 `맨몸 스플릿 스쿼트`다.

### 4.4 시간 배분 — `sessionTimes(s, kept, si)`
- 웜업(15%, 5~12분 clamp) / 쿨다운(10%, 4~8분) / 유산소(유산소날 32% 아니면 12%) / 운동별 시간이 **정확히 `state.sessionMinutes` 합**이 되게 배분. 30/45/60/90/120 모두 합 정확 검증됨.
- 유산소 시간은 `goalMods().cardioMul`을 곱한 뒤 **`cardioTarget`으로 기억**해두고, 반올림·"운동당 최소 3분" 보정에 깎이면 본 운동에서 되돌려 받는다. (이 보정이 없으면 목표별 유산소 증가가 근력 날에서 통째로 사라짐)

### 4.4b 사용자 목표 반영 — `goalMods()` (우선순위 가중, 2026-07 개편)
- `GOALS = ['다이어트(체중감량)','근육 만들기','체력 향상','자세교정','현 상태 유지']`.
- **`state.profile.goals`는 순서 있는 배열이고 최대 2개** — index 0 = 1순위, index 1 = 2순위. 온보딩에서 먼저 고른 순서가 우선순위이고, 칩에 `1순위`/`2순위` 뱃지(`.rankbadge`)가 붙는다. **목표 직접 입력(기타/`goalOther`)은 제거됨.**
- 목표별 효과는 `GOAL_FX` 테이블에 있고, 가중치 `GOAL_W=[1.0, 0.5]`로 **1순위는 그대로, 2순위는 절반만** 반영한다.

| 목표(GOAL_FX) | set | reps | cardio | rest | posture |
|---|---|---|---|---|---|
| 근육 만들기 | 1 | 0.85 | 0.85 | 60~90초 | — |
| 다이어트(체중감량) | 0 | 1.2 | 1.3 | 30~60초 | — |
| 체력 향상 | 0 | 1.0 | 1.4 | 45~75초 | — |
| 자세교정 | 0 | 1.0 | 1.0 | (없음) | true |
| 현 상태 유지 | 0 | 1.0 | 1.0 | (없음) | — |

- 합성: `repsMul *= 1+(fx.reps-1)*w`, `cardioMul *= 1+(fx.cardio-1)*w` (곱셈, 편차에 가중치). 예) 다이어트 1순위+체력 2순위 → cardioMul 1.3×1.2=1.56.
- `setDelta`: **근육 만들기가 1·2순위 중 어디든 있으면 +1** (모 아니면 도인 값이라 순위 무관).
- `rest`: 우선순위 높은 목표부터 `fx.rest`가 있는 첫 목표를 채택(1순위 우선). 없으면 60~90초.
- `repsMul`은 **횟수형(`12회`, `10회/측`)에만** 적용(`goalReps`). 유지형(`30–45초`)·범위형(`20–30초/측`)은 건드리지 않는다. 상한 8~20회.
- `recReps(slot,r)` = `goalReps(defReps(slot,r))`. **`effVol`·`volEditor`(추천값 리셋)·일정 요약(mini-ex) 세 곳 모두 이걸 써야** 표시가 어긋나지 않는다.
- **단일 목표로도 플랜이 선택 시간을 꽉 채운다**: `sessionTimes`가 T를 warm/cool/cardio/main에 비례 배분하므로 `total`은 항상 T. 목표는 기본 슬롯을 **빼지 않고** 세트/횟수/유산소만 조정하며, 자세교정은 교정 슬롯을 **추가**만 한다(자세교정 단독 = 가장 많은 슬롯).

#### 자세교정 = 세부 증상 선택식 (증상 1개 = 버튼 1개, 2026-07)
- "자세교정" 목표를 고르면 **세부 증상**을 고른다(`state.profile.postureTypes`, 온보딩 목표 아래 버튼, 복수 선택). **가운데점 금지 규칙 때문에 증상 1개당 버튼 1개**로 분리:
  - `neck` 거북목 / `shoulder` 말린 어깨 / `back` 굽은 등 / `pelvis` 골반 전방경사
- 근거: 상부/하부 교차 증후군(Janda). UCS 교정운동 RCT(PMC7692548), LCS NASM RCT(PMC12959714).
- 슬롯 태그가 `opt:'posture:neck|shoulder|back|pelvis'`로 나뉜다. `slotActive`가 목표 ON + 해당 증상 선택 시에만 노출. 증상 미선택이면 `ptypes()`가 상체 3종(`neck`,`shoulder`,`back`) 기본.
  - upper 슬롯: 거북목 교정(S1·S6, S10 집중일), 흉추 신전(S10), 자세교정 종합(S5).
  - pelvis 슬롯: 골반 정렬 (전방경사 교정)(S0·S3) + S3 쿨다운에 고관절 굴곡근 스트레칭.
  - 항상 노출되는 자세교정 (말린 어깨)·후면 삼각/능형근·자세교정 (하부 승모)는 근비대 가치가 있어 유지(유형 무관, 목표 시 세트+1).
- ⚠️ **다루지 않는 것**: 척추측만 등 구조적 변형·통증 동반 자세는 앱이 교정하지 않는다(온보딩에 명시). 하지 정렬(X/O다리·평발)도 근거 애매해 제외.
- ⚠️ **자세=통증 아님**: 자세 변형은 무증상자에게도 흔하며 병·통증을 뜻하지 않는다(문헌 공통). "고쳐야 한다"가 아니라 "도움될 수 있다"로 표현.

#### 자세교정 판정 — 단일 소스로 통합됨 (2026-07)
- 지금 "자세교정"은 **상·하부 교차 증후군(말린 어깨·거북목·굽은 등·골반 전방경사)**을 다룬다(증상 4종 neck/shoulder/back/pelvis, §4.4b).
- ~~정의가 `GOALS`/`_POSTURE_ROLE` 정규식/`opt:'posture'` 세 군데에 흩어져 있음~~ → **통합 완료**. 이제 **슬롯 데이터가 유일한 소스**다. role 이름 정규식(`_POSTURE_ROLE`, 죽은 패턴 `페이스 풀` 포함)은 **폐기**했다. 되살리지 말 것.
- **판정 헬퍼 두 개**(`effVol`·`slotActive`가 공통으로 사용):
  - `postureSym(sl)` = `opt:'posture:xxx'`의 증상(`neck|shoulder|back|pelvis`) 또는 `null`.
  - `isPostureSlot(sl)` = `sl.posture===true || postureSym(sl)!=null` → **자세교정 목표면 세트 +1** 대상(`effVol`).
- **슬롯 데이터 표기(둘 중 하나로만 마킹)**:
  - **항상 노출 + 목표 시 세트 +1** → `posture:true`. 대상: `자세교정 (말린 어깨)`, `후면 삼각/능형근`, `자세교정 (하부 승모)`(**S4·S7 모두**, 2026-07 통일). 근비대 가치가 함께 있어 항상 남긴다.
  - **증상 선택 시에만 노출** → `opt:'posture:neck|shoulder|back|pelvis'`. 대상: `거북목 교정`(neck), `자세교정 종합`(shoulder), `흉추 신전`(back), `골반 정렬`(pelvis).
  - ⚠️ **새 교정 슬롯은 반드시 이 둘 중 하나로 태그**할 것. role 이름으로 판정하지 않으므로 태그가 없으면 교정으로 인식되지 않는다.
- 노출 여부는 `slotActive(sl)` 한 곳에서만 판단한다(`postureSym` 경유). `sessionExercises`·`cautionedByInjury`·`compoundsInPlan` 모두 이걸 거친다.

### 4.5 웜업/쿨다운 — **동적 생성** (하드코딩 아님)
- 정의 위치: `const SESSIONS=[...]` 바로 뒤. `sessionCat(s)`, `isPullDay(s)`, `buildWU(s)`, `buildCD(s)` → `SESSIONS.forEach(s=>{ s.warm=buildWU(s); s.cool=buildCD(s); })`.
- **핵심 규칙: 한 항목 = 한 동작.** "전신 관절 풀기" 같은 결합/묶음 항목 **금지**. 각 부위를 개별 항목으로.
- **웜업 순서**: ① 가벼운 유산소(체온) → ② 전신 부위별 단일 동작 순서 **목 → 어깨 → 팔 → 허리(상체 회전) → 옆구리 → 고관절(레그 스윙) → 허벅지(니 하이 마치) → 종아리(카프 레이즈) → 손목 → 발목** → ③ 그날 부위 활성화(상체: 밴드 풀 어파트 + [당기기날엔 턱 당기기(자세교정+거북목 목표일 때만, 위 참조) / 밀기=벽 팔굽혀펴기]; 하체: 바디웨이트 스쿼트 + 밴드 사이드 스텝; 그 외(`full`·`cardio` — 유산소 포함, 같은 else 브랜치): 바디웨이트 스쿼트 + 밴드 풀 어파트 + 힙 서클).
  - **턱 당기기(당기기 날 활성화 항목)는 자세교정+거북목 목표일 때만 노출**한다(2026-07). `buildWU`가 `{n:'턱 당기기', …, posture:'neck'}`로 태그하고, 렌더 시 `wuResolve`가 `goalMods().posture && ptypes().includes('neck')`일 때만 남긴다. 다른 부위 태그도 같은 방식으로 게이팅 가능. **모든 사용자 고정으로 되돌리지 말 것**(과거 개인 요청 잔재였음).
- **쿨다운**: 부위별 단일 정적 스트레칭. 그날 부위 위주로 순서·시간 가중(M{} 딕셔너리: neck/shoulder/triceps/chest/back/side/glute/quad/ham/calf). 유산소날은 앞에 "호흡 정리 걷기" 추가.
- **그날 부위 중심 원칙**: 상체날=상체 위주, 하체날=하체 위주, 유산소날=유산소 적합 동작 위주. 단, 다른 부위를 빼는 게 아니라 **중심만 이동**(전신은 항상 커버).
- **턱 당기기(Chin Tuck)는 유지형(초)** — `wuTiming`이 횟수형으로 오인하지 않게 주의.
- 항목 태그 구조: `{n:이름, a:분량, d:설명, eq?:[장비], floor?:bool, alt?:{n,a,d}}` 또는 `{cardio:true}`.
- `wuResolve(arr)`: eq/floor로 필터(안 되면 alt로 대체, 그래도 안 되면 드롭).
- `cardioMachines()`/`cardioChips()`/`pickCardio(id)`: 유산소 기구 선택 버튼(state.cardioChoice).
- `listBody(arr)`: 두 줄 렌더 — `.wu-name`(이름 · 분량) + `.wu-desc`("- 설명").

### 4.6 웜업/쿨다운 자동 가이드 플레이어
- `wuTiming(it)`: `a` 문자열 파싱 → `{mode:'hold'|'reps', reps, dur}`. 초/분→hold(초); 회/걸음→reps(회, 템포 ~2초/회, 걸음 ~0.7초); 좌/우 각각·앞·뒤·각각→2배; ×N 세트 곱; hold clamp 15~150초, reps clamp 12~90초; cardio→300초.
- `wuBuildItems(arr)` = wuResolve 후 wuTiming 매핑. `fmtClock(sec)`→"m:ss".
- 모듈 변수 `let wuP=null, wuTimer=null;`.
- `wuStart(phase,dateStr)`: 아이템 생성, `wuP={phase,dateStr,items,idx,t0,beganAt}`, `setInterval(wuTick,200)`, renderToday.
- `wuTick()`: `#wup-fill` 너비 / `#wup-count` / `#wup-rep`(횟수는 전체 시간 동안 **균일 증가**) 갱신, 경과≥dur이면 자동 다음.
- `wuNext()`=건너뛰기. `wuFinish()`=stepMs 기록 후 wuP 해제, `doneStep(dateStr,phase)`.
- `wuPlayerCard()`: 큰 타이머 + 횟수 라인 + 진행바 + 점(dots) + [다음 →]/[웜업|쿨다운 마침].
- `wuStepCard(phase,s,dateStr,active,done,mins)`: 시작 전 카드([웜업|쿨다운 시작 →]).
- `renderSessionCard`에서 'warm'/'cool' 단계: wuP 활성이면 `wuPlayerCard()`, 아니면 `wuStepCard()`.
- **쿨다운 마침 → 운동 완료로 이어짐**(doneStep이 모든 단계 완료 시 그날 완료 처리).

### 4.7 본운동 단계별 타이머
- `stepShell`: 시작/완료 타이머. 미시작→[시작 →](`startStep`) → 진행중→[완료 · live-timer](doneStep이 stepMs 기록) → 완료→"✓ 완료 · N분 M초" + 다시 열기(`resetStep`).
- 전역 `setInterval(1s)`가 `.live-timer`(data-start) 갱신. `fmtDur(ms)`→"N분 M초".
- **세트 무게 입력 = −/+ 스텝퍼**(`wtCell`/`adjWeight`). 직접 타이핑도 되지만 폰트는 **반드시 16px 이상**이어야 iOS가 입력 포커스 시 화면을 자동 확대하지 않는다(과거 14px이라 확대 불편 있었음). 무게 승계는 프리필이 담당(아래 참조).
- **세트 수 ± 스텝퍼(`volEditor`→`chgSets`)**: `chgSets`에 넘기는 기준값은 **화면 표시값 `effVol().sets`(=`v.sets`)**여야 한다. `slot.sets`(raw)를 넘기면 목표(+1)·초보자(−1) 보정이 반영된 표시값과 어긋나 첫 탭이 값을 건너뛴다(2026-07 수정). 되돌리지 말 것.
  - **증량 단위는 기구별(`weightStep(eq)`)**: 덤벨 0.5 / 바벨·스미스·핵 스쿼트 2.5(원판) / 케틀벨 4 / 핀 스택 머신·케이블 5. 근거는 실제 무게 구성 조사(원판 최소쌍 2.5kg, 미터법 스택 5kg 등). 운동의 `eq` 배열로 판정한다.
  - **길게 누르면 연속 증가**(`wtHold`, `onpointerdown`): 첫 탭 1회, 계속 누르면 420ms 후부터 가속(×0.6, 하한 45ms). `pointerup`/`pointercancel`/`pointerleave`에서 멈춘다. 멀리 갈 때 편하도록.
  - **세트 칸 배치 = 브릭(`setBasis`)**: 1~3세트는 한 줄, 4+는 두 줄 균형(4→2+2, 5→3+2, 6→3+3, 7→4+3, 8→4+4). `cellBasis`와 달리 4를 2+2로 나눔.
  - **프리필(`.wt.pre` 흐리게, 2026-07 개선)**: 미저장 칸은 `wtCell`이 **`seedWeight`(같은 세션의 이전 세트 → 없으면 지난 기록) → `defaultWeight`(기구 기본값)** 순으로 미리 채운다. 즉 1세트에 60을 넣으면 2세트가 60(흐리게)으로 승계된다(자동 증량 아님 — 편집/±로 `pre` 해제되며 저장). ⚠️ `adjWeight`의 `isNaN` 분기는 프리필 때문에 거의 안 타므로, 프리필 로직을 `wtCell`에 두는 게 정답(예전엔 프리필이 항상 기본값이라 `seedWeight`가 죽은 코드였음).
  - **기본값 = 기구의 가장 가벼운 상태**: 머신·케이블 5 / 바벨 20(빈 봉) / 스미스 15 / 덤벨 남 5·여 3 / 케틀벨 4(남녀 공통). ⚠️ `wByGender`는 성별 저장값 `'female'`(GENDERS)로 판정 — `'여'` 같은 값과 비교하지 말 것. `exCat(name,eq)`로 5개 카테고리(machine/barbell/smith/dumbbell/kettlebell) 분류. 사용자가 설정 "무게 기본값"에서 카테고리별로 직접 조절(`state.wDefaults`, 편집 단위 `_WDEF_STEP`).
  - **무게 칸에 "kg" 단위 라벨을 넣지 않는다.** 3칸(3세트 한 줄)일 때 폭이 좁아 "100"·"22.5"가 잘렸던 문제 때문. 숫자만 가운데 정렬로 표시하고, 단위는 "지난 기록 … kg" 힌트와 맥락으로 전달(`aria-label="무게(kg)"`).
  - 일반: `.setcell.wtc{flex:1 1 118px}`로 폭이 좁으면 자동 줄바꿈(폰에서 보통 2칸씩). 좌/우(측면당): 헤더 칸수에 맞춰 `flex:1 1 0`로 줄바꿈 없이 균등 분할.
- **완료된 운동에 취소선(line-through) 금지** — 색만 muted 처리.
- ⚠️ **맨몸 슬롯 로그 키는 문자열 `"Nb"`**(예 `"0b"`, `"10b"`) — `sessionExercises`가 `key:oi+'b'`로 만든다. `renderSetLog`이 이 키를 onclick에 넣을 때 **반드시 따옴표로 감쌀 것**(`toggleSetDone('${date}','${oi}',${k})`). 안 감싸면 `0b`가 잘못된 숫자 리터럴이라 SyntaxError로 점 토글이 통째로 죽는다(2026-07 수정, DEFAULT_EQUIP=[]로 첫 진입이 맨몸이라 전면 영향이었음).
- **동기화 push는 디바운스(500ms)**: `touch()`가 `persist()`(120ms)와 별개로 `Sync.push()`를 500ms 트레일링으로 미룬다. 무게 롱프레스(`wtHold`, 45ms 가속)가 초당 수십 회 전체 문서 쓰기를 하던 것 방지.
- **웜업 플레이어(`wuTimer`/`wuP`)는 `switchTab`·`resetAll`에서 정리**한다 — 안 하면 백그라운드 인터벌이 살아 초기화된 state에 stray 완료를 기록.

- **버튼 그룹을 감싸는 "박스"(테두리+배경+패딩 컨테이너) 금지.** 버튼 각자의 테두리는 유지하되, 여러 버튼을 통째로 감싸는 네모는 없앰.
  - 과거 버그: 온보딩 행 `class="obchips seg"`가 별도 `.seg` 박스 스타일에 걸려 감싸는 네모 생성됨 → `.obchips.seg{background:none;border:none;padding:0;margin-bottom:0}`로 무효화. `.seg`(이메일 토글)·`.modeseg`(일정 토글)·`.ob-inline`(러닝 입력)의 감싸는 박스도 제거함.
- 다중 선택 칩은 `.obgrid`(2열 균등 너비). 성별·내외국인·본인인증·근력/러닝 유무 등 이지선다형은 `.obchips.seg`(한 줄 균등 너비).
- 폰트 Pretendard. 다크모드는 `prefers-color-scheme` 자동. 테두리 변수(라이트 `--line:#D8D3C6 --line-2:#E3DFD3`, 다크 `--line:#3A3F47 --line-2:#31363D`).
- **컬러 = 팬톤 기반** (2026-07 교체). 액센트는 **팬톤 62x 램프**(같은 색상군의 밝기 7단계)라, 나중에 테마 컬러 교체 기능을 만들 때 이 램프만 갈아끼우면 라이트·다크·틴트가 함께 따라온다.
  - `--accent` 626 C `#285C4D` / 다크 624 C `#789F90`, `--accent-ink` 627 C `#13322B` / 다크 623 C `#9AB9AD`, `--accent-soft` 626 C 12% 틴트.
  - `--sand` 7556 C / 다크 7407 C, `--danger` 7599 C / 다크 7416 C, 심박존 334·7489·142·7577·7619 C.
  - **`--on-accent`** = 액센트 배경 위 글자색(라이트 `#FFFFFF`, 다크 627 C). 다크에서 흰 글자는 대비 2.9:1로 WCAG AA 미달이라 도입함. **액센트 배경을 쓰는 요소는 반드시 `color:var(--on-accent)`** (하드코딩 `#fff` 금지).
  - **중성색(배경 `#F5F4EF`·흰색·검정)은 팬톤으로 바꾸지 말 것.** 최근접이 Cool Gray 1 C(ΔE 6.0)라 배경이 회색으로 어두워진다.
- **레이아웃 폭**: `.wrap`·`.nav-inner` `max-width:430px`(폰 폭). 넓은 화면·미리보기에서도 폰처럼 가운데 정렬(모바일 전용 앱). 640px 등으로 넓히지 말 것.
- ⚠️ **사용자 자유입력은 반드시 `esc()`로 이스케이프 후 innerHTML에 넣는다** — 닉네임·이름·부상/지병 "기타"·구글 name/email 등. 안 하면 `<`·`"`·`&`가 든 닉네임이 UI를 깨거나 자기-XSS가 된다(2026-07 전면 적용). 프리셋 목록값(부상/목표 등)은 앱 데이터라 불필요. 이중 이스케이프 주의(한 값은 한 번만). 검증: 특수문자 프로필로 전 화면 렌더 후 원시 `<img …>` 노출 0.
- **한글 문장부호 규칙**: 명사형 어미(~기/~하기 등)는 마침표 없음. `~다/~요`는 마침표 유지.
- **텍스트 줄바꿈 규칙 (전역 · 모바일 기준)** — 긴 문구는 다음 규칙으로 보기 좋게 끊는다. 핵심 헬퍼 **`smartBreak(text, cap)`**(cap=한 줄에 들어가는 대략 글자 수, 기본 26~27). `periodLines()`=`smartBreak(t,27)`(문단), `fmtCue()`=절 분리 후 `smartBreak(c,26)`(운동 큐/유산소). 웜업/쿨다운 항목 설명은 `wuDesc()`가 수동 `|` 마커(nowrap 조각)로 처리.
  1. **마침표(. ) 뒤에서 줄바꿈** — 문장 단위로 나눔.
  2. **쉼표 줄바꿈은 조건부**: ①문장이 한 줄(cap)을 넘칠 때만, ②쉼표 앞부분(part1)이 한 줄에 들어갈 때만(≤cap), ③쉼표가 문장의 35% 이후일 때만 → 그 쉼표에서 줄바꿈(중앙에 가장 가까운 것). **짧으면 그대로 한 줄**, **너무 길어 part1이 한 줄을 못 넘으면 억지로 안 끊고 자연 줄바꿈**(예: 안전 안내문의 긴 문장). 조기 쉼표(나열 "무릎, 어깨")도 안 끊음.
  3. 🚫 **`·`(가운뎃점) 전면 금지 — 예외 없음.** 사용자에게 보이는 어떤 텍스트에도 `·`를 쓰지 말 것. **새 문구를 만들 때도 마찬가지.**
     - 나열은 `/` 또는 쉼표, 라벨과 값의 구분은 `<span class="vdiv"></span>`(회색 세로선) 또는 괄호, 문장 구분은 쉼표나 마침표를 쓴다.
     - 데이터(`cue`, 웜업 `d`)의 `·`는 `fmtCue`/`smartBreak`/`wuDesc`가 쉼표·불릿으로 자동 변환하므로 화면에 안 나온다. `cardioGuide`의 `note`도 이제 `replace(/\s*·\s*/g,', ')`로 전역 변환한다(2026-07, raw 렌더 경로 방어). **그래도 raw로 innerHTML에 넣는 새 경로를 만들 땐 변환을 반드시 거칠 것.**
     - 확인 방법: 브라우저에서 `document.createTreeWalker(document.body, NodeFilter.SHOW_TEXT)`로 `·`를 포함한 텍스트 노드를 훑는다(`<script>`/`<style>`/SVG 제외). **0건이어야 한다.**
     - 안전 원칙(`safety-note`)의 불릿도 `·`에서 `-`로 바꿨다. 되돌리지 말 것.
  4. **어절(단어) 중간에서 안 끊기**: 전역 `word-break:keep-all`. 불릿("- ") 줄은 flex 행잉인덴트로 둘째 줄을 텍스트 첫 글자에 맞춤.
  5. **운동 큐/유산소 설명**: `" · "`·`" — "`(양옆 공백)를 절 구분으로 보고 `- ` 불릿으로 분리, 절 안은 쉼표/마침표에서 끊음. 새 문구 작성 시에도 위 규칙을 따를 것.

2. **"바닥 운동 제외" 시 일부 슬롯이 빈다.** (부분 해결) S5·S2의 코어 안정/측면 슬롯에 **서서 하는 로디드 캐리**(파머스 캐리 `eq:['dumbbell'/'kettlebell']` 40초, 수트케이스 캐리 30초/측)를 `options`로 넣어, 장비 보유자는 바닥 off여도 코어 3패턴이 유지된다. 맨몸 fb(버드독/사이드 플랭크)는 무장비 폴백으로 남김. **남은 과제**: 하체·기타 세션에도 바닥 동작만 있는 슬롯이 있으면 같은 방식(서서 하는 장비 대안)으로 보강할 것. ⚠️ 옵션이 `reps`를 덮어쓰면 좌/우(측면) 판정은 `slot.reps`가 아니라 `defReps(slot,r)` 기준이어야 한다(캐리 40초는 좌/우 없음).
3. **지병(conditions)은 운동 선택에 영향 없음** — 경고 문구만 나옴.
4. ~~어깨 부상자는 수직 밀기 슬롯이 빈다~~ → **해결됨**(caution 재설계: 제외가 아니라 후순위라 슬롯이 비지 않음. 2026-07-13 실측 확인).
5. `POSE_SVG` 4종의 이름 crop(§4.8) — 새 아이콘 받으면 제거.
6. **메디신볼은 선택해도 쓰이는 운동이 없다.** (케틀벨은 S5·S2 코어 캐리에서 사용됨)
7. **`러닝 경험`·`목표 체중`은 여전히 표시만 되고 플랜에 반영되지 않는다.**

### 해결됨 (되돌리지 말 것)
- ~~사용자 목표(goals) 미반영~~ → **`goalMods()`로 반영 완료**(§4.4b). 세트·반복·유산소·휴식이 목표별로 달라진다.

## 9. 작업 시 반드시 지킬 것 (요약)
1. 한국어로 소통. 명사형 어미 마침표 규칙 준수.
2. **불특정 다수용 앱.** 특정 개인의 신체 조건·일정·장비를 전역 기본값으로 넣지 말 것. 부상 제약은 사용자가 고른 값에만, `CAUTION_EX`로 **후순위 배치**(제외 아님, §3). **문서·코드·커밋 메시지에 실명이나 의료정보를 쓰지 말 것**(저장소는 공개).
3. 장비 게이팅: 선택 안 한 장비는 운동생성·설명·웜업/쿨다운·isBW 무료목록 어디에도 안 나오게. **기구가 하나도 없으면 맨몸 운동은 자동 ON이며 해제 불가**(설정·온보딩 양쪽).
4. 웜업/쿨다운: 한 항목=한 동작, 정식 명칭만, 전신 커버+그날 부위 중심, 유지형/횟수형 구분.
5. `index.html` 직접 편집 후 `<script>` 문법 확인(`node --check`). 배포는 `git pull --no-edit` → add → commit → push.
6. grid 레이아웃은 wkhtmltoimage로 시각검증 불가(실제 브라우저는 정상).
7. 커밋·푸시 전 운영자 확인. 자율 배포 지양.
8. 저장소 전체가 공개 배포됨 → 비밀키·PAT·민감정보·잡파일 커밋 금지(§1 참고).
